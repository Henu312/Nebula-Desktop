# Nebula Desktop 系统设计文档

## 1. 设计目标

Nebula Desktop 是一个 Windows 10 / Windows 11 桌面增强层，目标是在不替代 Windows Shell 的前提下，提供更现代、更高效、更稳定的桌面交互体验。

核心目标：

- 提供 mac 风格 Taskbar、Launcher、Control Center、Notification Badge。
- 保留 Explorer、System Tray、Notification Center、Window Manager 等 Windows 原生能力。
- 优先保证微信、QQ、企业微信、钉钉、VSCode、JetBrains IDE 等常用软件兼容性。
- 空闲 CPU 占用低于 1%，内存占用低于 200MB，启动时间低于 2 秒。

明确不做：

- 不替换 `explorer.exe`。
- 不做 Shell Replacement。
- 不 Hook Explorer 内部实现。
- 不修改 Windows XAML Tree。
- 不接管 Windows Notification Center。
- 不读取聊天内容、通知正文或隐私数据。

## 2. 总体架构

系统采用 Overlay Architecture：

```text
Frontend: React + TypeScript
├─ Taskbar UI
├─ Launcher UI
├─ Control Center UI
├─ Notification Badge UI
├─ Dock UI（Phase 2）
└─ Widget UI（Phase 3）

        ↓ Tauri IPC

Backend: Rust
├─ Window Manager
├─ AppBar Manager
├─ App State
├─ System API Layer
├─ Notification Watcher
├─ Config Manager
└─ Storage Layer

        ↓ Windows Native APIs

Windows
├─ Win32
├─ WinRT
├─ Core Audio
├─ WLAN API
├─ Bluetooth API
└─ Shell APIs
```

前端只负责 UI、交互状态和动效，不直接访问系统能力。系统能力统一由 Rust 后端封装，通过 Tauri Commands 暴露给前端。

## 3. 技术选型

| 层级 | 技术 | 用途 |
|---|---|---|
| 桌面框架 | Tauri 2.x | 桌面窗口、IPC、打包、系统集成 |
| 前端框架 | React + TypeScript | UI 组件、状态驱动渲染 |
| UI 样式 | TailwindCSS | 快速构建一致的现代 UI |
| 动画 | Framer Motion | Hover、展开、切换、Launcher 动效 |
| 后端 | Rust | Windows API 调用、状态管理、后台服务 |
| 存储 | SQLite + JSON Config | 固定应用、配置、最近记录、Workspace |
| 系统 API | Win32 / WinRT / Core Audio / WLAN | 窗口、音量、网络、蓝牙、AppBar |

## 3.1 Windows 版本兼容基线

Nebula Desktop 必须同时支持 Windows 10 和 Windows 11。设计基线以 Windows 10 可用能力为准，Windows 11 视觉和交互能力作为增强层处理。

最低目标环境：

- Windows 10 22H2 x64。
- Windows 11 23H2 / 24H2 / 25H2 x64。
- WebView2 Runtime。
- MSVC x64 运行环境。

兼容策略：

- 核心窗口管理使用 Win32 API，避免依赖 Windows 11 独占能力。
- AppBar、窗口枚举、前台窗口、窗口激活、音量控制均按 Windows 10 可用 API 设计。
- WinRT 能力必须做运行时能力检测；Windows 10 上不可用或行为不一致时，回退到打开 Windows 原生设置页。
- 毛玻璃、云母、圆角等视觉效果必须有降级方案；Windows 10 上优先使用透明、半透明、阴影和普通圆角，不强依赖 Windows 11 Mica。
- 多屏 DPI 以 Windows 10 Per-Monitor DPI 行为作为兼容底线。
- 所有系统设置入口必须分别验证 Windows 10 和 Windows 11 的 URI 可用性。

平台能力分级：

| 能力 | Windows 10 | Windows 11 | 策略 |
|---|---|---|---|
| Tauri WebView2 | 支持 | 支持 | 必须能力 |
| Win32 Window Manager | 支持 | 支持 | 核心基线 |
| SHAppBarMessage | 支持 | 支持 | 核心基线 |
| Core Audio | 支持 | 支持 | 核心基线 |
| WLAN API | 支持 | 支持 | 核心基线 |
| WinRT Bluetooth | 需检测 | 支持更稳定 | 不可用时跳转设置 |
| Mica / Windows 11 材质 | 不支持 | 支持 | 仅作为增强 |
| 圆角系统窗口风格 | 不完整 | 支持更好 | 前端自绘圆角兜底 |

## 4. 进程与窗口模型

### 4.1 主进程

Tauri Rust 后端作为主进程，负责：

- 初始化配置。
- 创建主 Overlay 窗口。
- 注册 IPC Commands。
- 启动后台状态监听器。
- 管理窗口位置、尺寸、显示层级。
- 管理系统 API 调用。

### 4.2 前端窗口

MVP 阶段建议使用一个主 WebView 窗口承载所有核心 UI：

- Taskbar 常驻。
- Launcher 作为浮层显示。
- Control Center 作为浮层显示。
- Notification Badge 附着在 Taskbar 图标上。

这样可以减少多窗口同步复杂度，先保证稳定性。后续如果 Control Center、Launcher 需要独立层级或跨屏幕定位，再拆分为独立 Tauri 窗口。

### 4.3 Overlay 行为

Overlay 窗口应具备：

- 无边框。
- 透明背景。
- 可置顶。
- 不抢占无关焦点。
- 支持根据显示器 DPI 调整尺寸。
- 支持 Top、Bottom、Left、Right 四种 Taskbar 位置。
- 支持 Windows 10 和 Windows 11 的不同窗口阴影、透明、焦点行为差异。

AppBar 区域通过 `SHAppBarMessage` 管理，用于为 Taskbar 预留屏幕边缘空间，避免应用窗口被遮挡。

## 5. 模块设计

## 5.1 Taskbar

职责：

- 显示固定应用。
- 显示运行中应用。
- 标记当前激活窗口。
- 标记最小化状态。
- 显示 Notification Badge。
- 触发 Launcher 和 Control Center。

前端模块：

```text
src/ui/taskbar/
├─ Taskbar.tsx
├─ TaskbarItem.tsx
├─ TaskbarAppIcon.tsx
├─ TaskbarStatusArea.tsx
└─ taskbar.types.ts
```

后端能力：

- 枚举当前顶层窗口。
- 获取窗口标题。
- 获取进程路径。
- 获取应用图标。
- 获取前台窗口。
- 激活窗口。
- 最小化、还原窗口。

后端 API：

```text
get_running_apps()
get_foreground_window()
activate_window(window_id)
minimize_window(window_id)
restore_window(window_id)
get_app_icon(process_path)
```

## 5.2 Window Manager

职责：

- 封装 Win32 窗口枚举。
- 过滤不可见、工具窗口、系统窗口。
- 建立窗口 ID 与应用信息映射。
- 监听前台窗口变化。
- 向前端推送运行应用状态。

使用 API：

```text
EnumWindows
GetWindowText
GetForegroundWindow
GetWindowPlacement
ShowWindow
SetForegroundWindow
GetWindowThreadProcessId
```

设计原则：

- 不高频轮询。
- 优先事件驱动。
- 必要轮询必须低频，并使用缓存去重。
- 不注入其他进程。

## 5.3 AppBar Manager

职责：

- 将 Nebula Taskbar 注册为屏幕边缘 AppBar。
- 根据用户配置调整位置。
- 多屏环境下记录所属显示器。
- DPI 变化时重新计算工作区。

使用 API：

```text
SHAppBarMessage
```

关键状态：

```text
position: top | bottom | left | right
monitor_id: string
thickness: number
dpi_scale: number
```

## 5.4 Launcher

职责：

- 快速搜索应用、文件、命令、最近项目。
- 支持固定应用。
- Phase 2 支持 Command Palette。

MVP 数据源：

- 固定应用配置。
- 开始菜单快捷方式。
- 最近文件记录。

前端模块：

```text
src/ui/launcher/
├─ Launcher.tsx
├─ LauncherInput.tsx
├─ LauncherResultList.tsx
├─ LauncherResultItem.tsx
└─ launcher.types.ts
```

后端 API：

```text
search_apps(query)
search_recent_items(query)
launch_app(app_id)
open_file(path)
```

MVP 搜索策略：

- 本地内存索引。
- 输入防抖。
- 不做全文文件内容索引。
- 不扫描大目录。

## 5.5 Control Center

职责：

- 提供 WiFi、蓝牙、音量、亮度、夜间模式、电源模式入口。
- 高级设置跳转 Windows 原生设置页。

MVP 优先级：

1. 音量读取与调整。
2. WiFi 状态读取与设置入口。
3. 蓝牙状态读取与设置入口。
4. 夜间模式和电源模式入口。
5. 亮度控制视设备支持情况实现。

后端 API：

```text
get_volume()
set_volume(value)
get_wifi_status()
open_wifi_settings()
get_bluetooth_status()
open_bluetooth_settings()
open_display_settings()
open_power_settings()
```

设计原则：

- 复杂或权限敏感设置优先跳转 Windows 原生设置页。
- 不强行重写系统控制面板。
- 对不支持的硬件能力返回明确状态。

## 5.6 Notification Badge

职责：

- 标记某个应用是否存在未读或通知状态。
- 不读取通知正文。
- 不解析聊天内容。
- 不接管通知系统。

可用信号：

- Tray Icon 状态变化。
- Window Flash 状态。
- Window Attention 状态。

MVP 策略：

- 第一阶段以 Window Flash / Attention 状态为主。
- Tray 观察作为后续增强。
- 对微信、QQ、企业微信优先建立兼容测试用例。

后端 API：

```text
get_notification_indicators()
subscribe_notification_indicator_changed()
```

## 5.7 Config Manager

职责：

- 读取和保存用户配置。
- 为前端提供默认配置。
- 管理配置版本迁移。

配置示例：

```json
{
  "taskbarPosition": "top",
  "dockEnabled": false,
  "blur": true,
  "theme": "dark",
  "launcherHotkey": "Alt+Space",
  "taskbarThickness": 56
}
```

配置原则：

- 用户可编辑配置存 JSON。
- 运行时状态不直接写入配置。
- 配置变更必须校验合法值。

## 5.8 Storage Layer

SQLite 存储：

- 固定应用。
- 最近应用。
- 最近文件。
- Workspace（Phase 2）。
- 搜索索引元数据。

MVP 表设计建议：

```sql
CREATE TABLE pinned_apps (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  path TEXT NOT NULL,
  icon_path TEXT,
  sort_order INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE TABLE recent_items (
  id TEXT PRIMARY KEY,
  kind TEXT NOT NULL,
  title TEXT NOT NULL,
  path TEXT NOT NULL,
  last_opened_at INTEGER NOT NULL
);

CREATE TABLE app_cache (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  path TEXT NOT NULL,
  icon_cache_key TEXT,
  updated_at INTEGER NOT NULL
);
```

## 6. IPC 设计

IPC 采用 Tauri Commands：

```rust
#[tauri::command]
fn get_running_apps() -> Result<Vec<RunningApp>, String> {
    todo!()
}
```

前端调用：

```ts
import { invoke } from "@tauri-apps/api/core";

const apps = await invoke<RunningApp[]>("get_running_apps");
```

IPC 设计规则：

- Command 名称使用 snake_case。
- 返回值必须可序列化。
- 错误统一返回结构化错误。
- 长耗时操作异步执行。
- 频繁状态更新使用事件推送，不用前端高频拉取。

错误结构：

```ts
type NebulaError = {
  code: string;
  message: string;
  detail?: string;
};
```

## 7. 状态管理

前端状态分三类：

| 类型 | 示例 | 管理方式 |
|---|---|---|
| UI 状态 | Launcher 是否打开、Control Center 是否展开 | React local state |
| 系统状态 | 当前运行应用、前台窗口、音量 | Tauri IPC + store |
| 用户配置 | 主题、Taskbar 位置、固定应用 | Config API + store |

建议在 MVP 使用轻量状态管理：

- 简单组件内状态用 React `useState`。
- 跨组件状态可使用 Zustand。
- 不引入复杂全局状态框架。

## 9. 性能设计

性能目标：

- 空闲 CPU < 1%。
- 内存 < 200MB。
- 启动时间 < 2s。

约束：

- 禁止高频轮询。
- 禁止 Explorer Hook。
- 禁止 DWM Hook。
- 禁止全局 DLL 注入。
- 禁止扫描全盘文件。

优化策略：

- 系统状态事件驱动。
- 对运行应用列表做缓存。
- 图标异步加载并缓存。
- Launcher 输入防抖。
- 大型系统能力按需初始化。
- UI 动效限制在 transform 和 opacity。

## 10. 安全与隐私

安全边界：

- 不读取聊天内容。
- 不读取通知正文。
- 不上传用户数据。
- 不注入其他进程。
- 不 Patch 系统 DLL。
- 不使用 undocumented API。

权限原则：

- 默认不要求管理员权限。
- 需要系统设置时跳转 Windows 原生设置。
- 所有本地数据仅存用户目录。

## 11. 多屏与 DPI

必须支持：

- Windows 10 和 Windows 11。
- 多显示器。
- 125%、150%、200% 缩放。
- 不同显示器不同 DPI。
- Taskbar 在指定显示器边缘停靠。

设计要求：

- 后端维护显示器列表和 DPI 信息。
- 前端尺寸使用逻辑像素。
- AppBar 注册时按物理像素和 DPI 转换处理。
- 显示器变化时重新布局。

## 12. MVP 范围

MVP 包含：

- Tauri + React 工程骨架。
- Taskbar 基础 UI。
- 固定应用管理。
- 运行应用枚举。
- 前台窗口标记。
- Launcher 基础搜索与启动。
- Control Center 基础 UI。
- 音量读取和调整。
- Notification Badge 初版。
- 配置读写。

MVP 不包含：

- Workspace。
- 插件系统。
- Widget。
- AI Command。
- Dock 高级动画。
- 自定义通知中心。
- 全文文件搜索。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| AppBar 与 Windows 原生任务栏冲突 | 窗口遮挡或布局异常 | MVP 先支持单显示器主屏，逐步扩展多屏 |
| 微信/QQ 通知状态不可稳定获取 | Badge 不准确 | 优先使用非侵入信号，提供兼容测试矩阵 |
| 多屏 DPI 计算复杂 | UI 错位 | 独立封装 Monitor/DPI 模块 |
| 系统 API 权限差异 | 某些控制不可用 | 不可用时跳转原生设置页 |
| Windows 10 与 Windows 11 视觉能力差异 | UI 效果不一致 | Windows 10 使用降级样式，Windows 11 开启增强材质 |
| 高频状态同步导致占用升高 | CPU 超标 | 后端缓存、事件推送、低频兜底 |

## 14. 验收标准

MVP 验收：

- 应用可在 Windows 10 22H2 启动。
- 应用可在 Windows 11 25H2 启动。
- Taskbar 可显示在 Top 或 Bottom。
- 能显示固定应用。
- 能枚举并显示运行中窗口。
- 能标记当前激活窗口。
- Launcher 可打开固定应用。
- Control Center 可显示并调整音量。
- 不替换 Explorer。
- 不接管 Notification Center。
- 空闲 CPU 低于 1%。
- 内存占用低于 200MB。
- 退出后 Windows 原生桌面恢复正常。
